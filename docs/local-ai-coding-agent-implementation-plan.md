# 《本地桌面 AI 编程 Agent MVP》开发任务拆解 + 迭代排期

## 1. 文档目标

本文基于 [local-ai-coding-agent-mvp-prd.md](D:\py\yuanbao_agent\docs\local-ai-coding-agent-mvp-prd.md) 与 [local-ai-coding-agent-mvp-tech-spec.md](D:\py\yuanbao_agent\docs\local-ai-coding-agent-mvp-tech-spec.md)，输出一份可直接用于执行的 MVP 开发拆解方案，目标是按 PRD 中的 P0 闭环优先交付，并为 P1 能力预留扩展位。

建议交付边界：

- MVP 必须覆盖：工作区选择、会话、搜索/读文件、命令执行、patch/diff、审批、Git 状态、自动验证、一轮自动修复、中断恢复、日志追溯。
- P1 作为“可并行插入但不阻塞 MVP 上线”的增强项：关键配置文件优先读取、忽略规则、设置页、时间线视图、会话摘要、错误聚合。

建议排期假设：

- 适用 3 到 4 人跨端小队。
- 角色建议：前端 1 人、Tauri/Rust 1 人、Python Runtime 1 人、测试/产品由团队兼职承担。
- 周期建议：`3 个 Sprint（每个 2 周）+ 1 周缓冲/发布准备`，共 7 周。

## 2. 工作流拆分

| 层 | 核心职责 | 主要输入 | 主要输出 | 关键接口/约束 |
| --- | --- | --- | --- | --- |
| 前端（React + TS） | 会话 UI、任务步骤 UI、Diff UI、命令输出 UI、设置页、审批交互 | 用户输入、Tauri 事件流、历史会话数据 | UI 状态、调用业务命令、审批动作 | 只调用业务级 API，不直接触碰底层工具 |
| Tauri Core（Rust） | 前端可信网关、sidecar 生命周期、IPC 参数校验、权限边界、事件转发 | 前端 invoke 请求、Python runtime 事件 | 标准化 RPC 请求、统一事件总线、宿主能力 | 必须完成路径越界校验、参数校验、runtime 可达性兜底 |
| Python Runtime | Agent 编排、上下文构建、工具调度、任务状态推进、自动修复逻辑 | Tauri 转发的业务请求、SQLite 数据、模型响应 | 任务执行结果、tool call、patch、command log、事件流 | 所有工具调用均由 runtime 内部发起 |
| 存储层（SQLite） | 会话、消息、任务、tool call、patch、command log、approval、config 持久化 | Runtime 写入事件、业务状态 | 可恢复历史、可追溯日志、任务状态快照 | schema 可迁移，支持异常退出恢复 |
| 工具层（Search/File/Patch/Shell/Git/Policy） | 代码检索、文件读取、补丁应用、命令执行、Git 信息获取、安全策略执行 | Runtime 调度、工作区配置 | 结构化工具结果、错误码、审批请求 | 路径限制、命令超时、风险命令拦截、dry-run 优先 |

## 4. 每个模块的开发任务清单

### E0. 工程骨架与通信链路

- [ ] 初始化 Tauri + React + TypeScript 工程结构，明确目录边界。
- [ ] 建立 Python sidecar 启动、健康检查、退出重启策略。
- [ ] 定义统一 JSON-RPC 2.0 envelope 与事件 envelope。
- [ ] 打通前端 `invoke` 到 Tauri，再到 Python runtime 的请求链路。
- [ ] 打通 Python runtime 到 Tauri，再到前端的统一事件流。
- [ ] 补齐 shared schema 或 DTO 定义，避免三端字段漂移。
- [ ] 建立开发态日志输出与本地调试开关。
- [ ] 建立最小 E2E 冒烟链路：发送消息，返回 `task_id`，收到 `task.started` 事件。

完成标志：

- 可以从 UI 发起一条消息，前端可收到 runtime 返回的任务状态与流式 token 事件。

### E1. 工作区、会话与任务模型

- [ ] 实现工作区选择、工作区保存与最近工作区列表。
- [ ] 建立 `workspaces / sessions / messages / tasks` 四张基础表及 migration。
- [ ] 实现 `workspace.open / session.create / session.get / session.list / message.send / task.get`。
- [ ] 前端完成会话列表、当前会话页、任务状态展示。
- [ ] 会话关闭后可恢复，支持按工作区查看历史。
- [ ] 为任务创建唯一 ID，并建立任务与会话的归属关系。
- [ ] 落地基础错误模型映射，区分前端提示、宿主错误、runtime 业务错误。

完成标志：

- 用户可以打开工作区、创建会话、发送消息、查看历史会话，并看到任务状态变化。

### E2. 事件流与任务状态机

- [ ] 实现 `queued/running/waiting_approval/completed/failed/cancelled` 状态机。
- [ ] 将长流程全部挂载到 `task` 维度，统一 task 生命周期。
- [ ] 定义并落地事件类型：`task.* / assistant.token / tool.* / command.output / patch.proposed / approval.*`。
- [ ] 前端订阅事件并按 `session_id/task_id` 过滤渲染。
- [ ] 将关键事件持久化到数据库或日志文件，支持异常恢复。
- [ ] 处理 runtime 异常退出、前端重载后的订阅恢复。
- [ ] 增加任务时间线基础组件，为 P1 时间线视图预留数据结构。

完成标志：

- 任一任务在执行中、等待审批、失败、中断后，前端都能稳定显示当前状态并可恢复查看。

### E3. 核心工具层

- [ ] 实现 `list_dir`，支持递归与最大深度限制。
- [ ] 实现 `search_files`，优先对接 `ripgrep`，覆盖文件名搜索与内容搜索。
- [ ] 实现 `read_file` 与 `read_file_range`，支持大文件分段读取与编码兜底。
- [ ] 实现 `run_command`，支持 stdout/stderr 流式输出、超时、退出码记录、手动终止。
- [ ] 实现 `apply_patch`，优先支持 dry-run，返回 patch 摘要与变更文件数。
- [ ] 实现 `git_status` 与 `git_diff`。
- [ ] 建立 `tool_registry` 与 schema 校验，统一工具入参与错误码。
- [ ] 落地 `tool_calls / patches / command_logs` 三类持久化。

完成标志：

- Runtime 能独立完成搜索、读文件、执行命令、生成 patch dry-run、读取 git 状态，并有结构化结果。

### E4. Agent 编排与上下文构建

- [ ] 实现 `orchestrator` 主循环，支持“规划 -> 工具调用 -> 汇总 -> 产出 patch/结论”。
- [ ] 实现 `planner`，输出结构化 plan steps。
- [ ] 实现 `context_builder`，聚合最近消息、已读文件、搜索结果、关键配置文件、最近命令输出。
- [ ] 加入 token 预算控制与上下文裁剪策略。
- [ ] 接入 Provider Adapter，先稳定支持单模型主路径。
- [ ] 约束工具调用来源，仅允许 orchestrator 经 `tool_registry` 调度工具。
- [ ] 为关键配置文件优先加载留钩子，便于 Sprint 3 插入 P1 能力。

完成标志：

- 用户输入“修复单测”后，runtime 能先生成计划，再调用工具完成一次完整分析，并返回后续动作。

### E5. Diff 审批与安全策略

- [ ] 实现 `policy_guard` 的路径越界检查。
- [ ] 实现命令风险分级、黑白名单与阻断模式。
- [ ] 支持三种审批策略：全审批、风险审批、宽松模式。
- [ ] 审批实体落库：`approvals` 表与 `approval.submit` 流程。
- [ ] 前端完成审批卡片、Diff 预览、风险说明、批准/拒绝交互。
- [ ] patch 应用前必须先 dry-run 并展示统一 diff。
- [ ] 写文件、删除文件、高风险命令、网络权限接入统一审批门控。
- [ ] 支持“本次允许”与“拒绝”基础策略，是否扩展“记住策略”可放 P1。

完成标志：

- 任一写操作和高风险命令都不能绕过审批链路，且用户能明确看到影响范围和当前任务阻塞状态。

### E6. 自动验证、失败回灌与恢复

- [ ] 为任务定义验证阶段，支持默认测试/构建命令执行。
- [ ] 将验证失败的 stdout/stderr 聚合回灌到模型上下文。
- [ ] 支持至少一轮自动修复循环。
- [ ] 实现任务取消：中断模型调用、中断命令、状态更新为 `cancelled`。
- [ ] 实现命令超时后的 kill 与状态落库。
- [ ] 保存中间 patch、命令日志、失败原因，支持后续恢复查看。
- [ ] 为恢复执行预留入口：重新进入同会话继续任务。

完成标志：

- 在一个真实失败测试场景中，Agent 能执行“运行测试 -> 修改代码 -> 审批 -> 验证 -> 失败回灌 -> 再修一轮”。

### E7. 配置、可观测性与发布验收

- [ ] 建立 `config_entries` 与配置覆盖规则：`session > workspace > global`。
- [ ] 实现基础设置项：默认模型、审批模式、命令超时、允许网络、UI 语言。
- [ ] 记录关键指标：任务耗时、工具调用次数、命令成功率、patch 成功率、用户中断率。
- [ ] 实现本地日志导出，便于排障。
- [ ] 建立错误码埋点与异常归类看板。
- [ ] 输出 MVP 验收用例集，覆盖至少 3 类真实仓库。
- [ ] 执行跨平台冒烟验证：Windows 优先，macOS 至少完成核心链路验证。

完成标志：

- 团队可以通过日志、任务记录和验收用例追踪一次任务从发起到完成/失败的全过程。

### E8. P1 体验增强

- [ ] 自动优先读取 `README`、`package.json`、`pyproject.toml`、`Makefile` 等关键文件。
- [ ] 支持 `.gitignore` 与自定义 ignore 配置。
- [ ] 完善任务时间线视图与错误聚合视图。
- [ ] 增加会话摘要，降低长会话上下文成本。
- [ ] 完善设置页与用户偏好持久化。
- [ ] 支持多模型配置但单任务仅使用单模型。
- [ ] 为 P2 的 MCP / LSP / 浏览器验证预留扩展位。

完成标志：

- 不改变核心闭环的前提下，显著提升使用效率与可理解性。

## 5. 建议迭代排期

### 方案建议

采用 `3 个双周 Sprint + 1 周缓冲`：

- Sprint 1：打通 M1
- Sprint 2：打通 M2
- Sprint 3：打通 M3 + 部分 P1
- Release Buffer：稳定性、性能、真实仓库回归、修边

### 迭代排期表

| Sprint | 周次 | 目标 | 主要交付 |
| --- | --- | --- | --- |
| Sprint 1 | 第 1-2 周 | 完成可用的本地聊天与工具壳 | E0、E1、E2 的主体；E3 中 `list_dir/search/read_file`；基础模型接入；历史会话与流式输出 |
| Sprint 2 | 第 3-4 周 | 完成闭环式代码修改 | E3 剩余工具；E4；E5 主体；patch dry-run、diff 展示、审批、命令执行、git 状态 |
| Sprint 3 | 第 5-6 周 | 完成自动验证与错误恢复 | E6、E7 主体；默认验证命令、一轮自动修复、中断恢复、配置、日志、指标；插入 E8 的关键文件优先读取与错误聚合 |
| Release Buffer | 第 7 周 | 质量收口与发布准备 | 真实仓库验收、跨平台回归、性能调优、风险项兜底、文档与演示材料 |

### 每个 Sprint 的退出标准

#### Sprint 1 退出标准

- 能打开工作区并发起会话。
- 能搜索文件、读取片段、流式返回模型输出。
- 会话和任务能持久化并恢复。
- 事件流在前端可稳定渲染。

#### Sprint 2 退出标准

- 能通过 Agent 发起代码修改请求。
- patch 必须先 dry-run 并展示 diff。
- 用户审批后可应用补丁。
- 可执行测试或构建命令并查看输出。
- Git diff 能反映本次变更。

#### Sprint 3 退出标准

- 验证失败后，Agent 能至少自动重试修复一轮。
- 用户可取消运行中的任务。
- 高风险操作全部走审批规则。
- 日志、配置、命令记录可追溯。
- 至少 1 个真实仓库完成端到端演示。

#### Release Buffer 退出标准

- 至少 3 个真实本地仓库、10 个任务完成验收。
- 至少 70% 任务实现“搜索 -> 修改 -> 验证 -> 展示结果”的闭环。
- 无阻断式崩溃。
- 写操作全部具备 diff 审批能力。

## 6. 关键依赖关系

| 依赖项 | 前置 | 后置 | 说明 |
| --- | --- | --- | --- |
| 统一 RPC / 事件协议 | E0 | E1/E2/E4/E5/E6 | 没有统一协议，三端联调成本会快速失控 |
| SQLite 基础 schema | E1 | E2/E3/E5/E6/E7 | 会话、任务、日志、审批都依赖持久化模型 |
| 任务状态机 | E2 | E5/E6/前端时间线 | 审批等待、中断恢复、自动修复都依赖明确 task 状态 |
| `search/read/run_command` | E3 | E4/E6 | Agent loop 最小可用工具集 |
| `apply_patch` dry-run | E3 | E5/E6 | diff 审批与安全写入的前置条件 |
| Policy Guard | E5 | 所有写入/命令路径 | 必须先有门控，才能开放 patch 与命令自动执行 |
| command log 持久化 | E3 | E6/E7 | 自动修复与排障都依赖完整命令输出 |
| Provider Adapter | E4 | E6/E8 | 一轮自动修复与未来多模型配置依赖统一适配层 |
| 配置分层 | E7 | E5/E6/E8 | 审批模式、超时策略、忽略规则都依赖配置体系 |

建议实施顺序严格遵循：

`E0 -> E1/E2 -> E3 -> E4 -> E5 -> E6 -> E7 -> E8`

## 7. 验收标准

### 功能验收

1. 用户可选择本地工作区并创建/恢复会话。
2. Agent 可在工作区内完成目录浏览、文件搜索、文件读取。
3. Agent 可生成补丁并以 diff 形式展示。
4. 写操作必须经过审批后才应用。
5. Agent 可运行测试或构建命令，并实时展示输出。
6. 验证失败时，Agent 可基于错误日志自动再修复至少一轮。
7. 用户可中断任务，且中断后历史记录可恢复查看。
8. Git 状态与变更可被查询并展示。

### 安全验收

1. 默认只允许访问当前工作区。
2. 工作区外路径访问必须被拦截或触发审批。
3. 高风险命令不能绕过审批。
4. 命令必须支持超时与手动终止。
5. patch 应用前必须可预览 diff，且保留日志。

### 非功能验收

1. 首次打开工作区后，目录可见时间不超过 3 秒。
2. 中小型仓库全文搜索响应时间不超过 2 秒。
3. UI 对工具状态更新的感知延迟不超过 500ms。
4. 异常退出后，可恢复到最近一次可用会话状态。
5. 关键任务、命令、审批、patch 均可追溯。

### 发布验收

1. 在至少 3 个真实仓库完成 10 个以上任务测试。
2. 至少 70% 任务可完成完整闭环。
3. 不出现阻断式崩溃。
4. 所有写操作路径均经过 diff 审批与记录。

## 8. 风险与缓冲建议

| 风险 | 影响 | 建议应对 | 缓冲建议 |
| --- | --- | --- | --- |
| Python sidecar 与 Tauri 的 stdio/事件链路不稳定 | 流式输出中断、任务卡死 | Sprint 1 优先打通链路并做长连接压测 | 第 2 周预留 1-2 天专项联调 |
| Windows/macOS 命令行为差异 | 命令执行与路径处理不一致 | 统一封装 shell service 与路径归一化 | 第 7 周保留跨平台回归 |
| patch 应用受 CRLF、编码、文件变更冲突影响 | patch 失败率高 | 强制 dry-run，失败时返回明确错误与回退建议 | Sprint 2 预留 1 天补丁兼容处理 |
| 模型上下文过载导致成本高、效果差 | 自动修复成功率低 | 严格做上下文裁剪与关键文件优先加载 | Sprint 3 预留 1 天调参 |
| 高风险命令识别不准 | 安全风险或误拦截 | 先用保守白名单，逐步放开 | 发布前补 1 轮安全 review |
| 自动修复循环不稳定 | MVP 闭环成功率不足 | 只承诺“一轮自动修复”，避免过度自动化 | 第 6-7 周重点回归此场景 |
| SQLite 并发与异常恢复不足 | 会话丢失、状态错乱 | 所有关键事件落库，减少仅内存态 | Sprint 1 完成后做异常退出测试 |
| 真实仓库差异大 | 验收通过率不稳定 | 提前准备 Python、Node.js、前端仓库各 1 个样本库 | 第 7 周至少 3 天真实仓库回归 |

### 缓冲原则

- 每个 Sprint 按有效产能预留 `15% - 20%` 作为技术缓冲。
- Release Buffer 不再新增 P0 范围，仅做稳定性修复与验收收口。
- 若 Sprint 2 结束时 E5 未闭环，必须冻结 E8，优先保证审批、安全、patch 主链路。
- 若 Sprint 3 中自动修复稳定性不足，可降级为“失败日志总结 + 用户确认后再次修复”，但不能牺牲审批与任务可追溯性。

---

## 9. 建议的里程碑映射

| 里程碑 | 对应史诗 | 输出 |
| --- | --- | --- |
| M1：可用的本地聊天与工具壳 | E0 + E1 + E2 + E3（部分） | 工作区、会话、搜索、读文件、流式响应、历史恢复 |
| M2：闭环式代码修改 | E3 + E4 + E5 | Agent 工具循环、patch、diff、审批、命令执行、Git 能力 |
| M3：自动验证与错误恢复 | E6 + E7 + E8（部分） | 自动验证、一轮自动修复、中断恢复、配置、日志、错误聚合 |

---

## 10. 结论

本项目的正确推进方式不是按技术层孤立开发，而是围绕 MVP 主闭环逐步打通：

`工作区/会话 -> 搜索/读取 -> Agent 计划 -> patch/diff -> 审批 -> 验证 -> 失败回灌 -> 恢复追溯`

建议以 `E0-E7` 作为 MVP 必做范围，以 `E8` 作为不阻塞上线的增强包执行。若团队资源有限，优先级必须始终服从闭环路径，而不是 UI 完整度或多模型能力。
