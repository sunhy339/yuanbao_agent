# 本地桌面 AI 编程 Agent MVP PRD

## 1. 产品概述

### 1.1 产品定位

本产品是一款面向单用户、本地使用的桌面 AI 编程 Agent，帮助开发者在本地代码仓库中完成代码理解、修改、验证与迭代，形成可控、可审查、可恢复的开发闭环。产品形态参考 `Codex` / `Claude Code` 的使用方式，但聚焦于本地单机开发体验，不追求平台化和多租户能力。

### 1.2 产品目标

- 支持用户在本地项目目录中发起自然语言任务，如“修复单测”“重构某模块”“补充类型定义”“定位构建错误”。
- Agent 能基于本地代码仓库执行核心循环：`搜索代码 -> 读取文件 -> 制定计划 -> 修改代码 -> 运行测试/命令 -> 展示 diff -> 用户确认`。
- 所有高风险操作具备用户可见、可审批、可中断能力。
- 保证本地优先，数据与工作区默认留在本机，降低云端依赖。
- 为后续扩展更强的代码理解、插件系统、多模型路由打下基础，但 MVP 阶段不引入重型基础设施。

### 1.3 成功定义

若产品可在真实本地项目中稳定完成以下任务，即视为 MVP 成功：

- 用户输入需求后， Agent 能准确搜索相关代码并读取关键文件。
- Agent 能生成补丁并以 diff 形式展示。
- 用户确认后，Agent 能应用补丁并执行验证命令。
- 当验证失败时，Agent 能基于错误日志进行至少一轮自动修复。
- 用户对整个过程保持可见和可控。

### 1.4 非目标

MVP 阶段明确不做以下能力：

- 多用户协作、账号体系、权限系统。
- 云端同步、远程执行、团队共享会话。
- 多 Agent 并行协作、DAG 编排、Workflow Engine。
- 插件市场、第三方扩展生态。
- 自动提交 PR、自动 push、CI/CD 联动。
- 向量数据库、复杂长期记忆系统。
- 面向企业的审计、租户隔离、计费系统。

## 2. 目标用户与典型场景

### 2.1 目标用户

目标用户为单人开发者或小团队中的个人开发者，具备本地开发环境和基础命令行能力，期望通过 AI 提升编程效率，但又不希望失去对代码与执行过程的控制。

主要用户类型：

- 独立开发者
- 后端工程师
- 前端工程师
- 全栈工程师
- 使用 Python / Node.js / Rust 等本地项目的技术人员

### 2.2 典型场景

典型使用场景包括：

- 修复失败测试：用户要求 Agent 定位并修复某个测试失败问题。
- 小范围重构：用户要求 Agent 调整函数拆分、重命名或抽取公共逻辑。
- 补充类型与文档：用户要求 Agent 为现有代码补充类型注解、注释或 README 内容。
- 排查构建错误：用户要求 Agent 根据编译或运行时报错定位问题来源。
- 实现局部功能：用户要求 Agent 在已有项目中新增一个简单接口、命令或页面逻辑。

---

## 3. 用户故事

- 作为开发者，我希望输入一句自然语言需求，Agent 就能在本地项目中搜索并定位相关代码。
- 作为开发者，我希望 Agent 在修改代码前先展示将要修改的 diff，而不是直接覆盖文件。
- 作为开发者，我希望 Agent 在修改后自动运行测试或构建命令，以验证改动是否有效。
- 作为开发者，我希望当验证失败时，Agent 能继续读取错误信息并尝试修复，而不是把问题留给我手动处理。
- 作为开发者，我希望所有 shell 执行、文件写入、删除操作都可配置为审批，以避免误操作。
- 作为开发者，我希望关闭应用后，下次还能查看历史会话、已执行命令和修改记录。
- 作为开发者，我希望 Agent 默认只访问当前工作区，而不能随意读写本机其他目录。
- 作为开发者，我希望在任务运行过程中随时停止 Agent，避免无休止执行或成本失控。

## 4. 功能范围

### 4.1 P0 必须交付

P0 为 MVP 核心闭环，必须具备。

#### 工作区与会话

- 选择本地项目目录作为当前工作区。
- 创建、查看、恢复会话。
- 聊天式输入任务需求。
- 展示当前任务执行状态。

#### 代码理解与检索

- 列出目录结构。
- 按文件名搜索。
- 按全文内容搜索，底层优先使用 `ripgrep`。
- 按片段读取文件内容。
- 能将搜索结果和已读文件纳入模型上下文。

#### 工具执行

- 运行 shell 命令并展示实时输出。
- 读取文件、写入文件、创建文件。
- 基于 patch/diff 的代码修改能力。
- 查看 Git 状态和 Git diff。

#### Agent 核心流程

- 支持简单计划生成。
- 支持工具调用循环：搜索、读文件、运行命令、修改、验证。
- 修改后可自动执行测试或构建命令。
- 验证失败后至少支持一轮自动修复。
- 支持中断当前任务。

#### 可控性与安全

- 写文件前展示 diff 并可审批。
- 命令执行前支持审批模式。
- 限制默认可访问目录为当前工作区。
- 命令超时与步数上限。
- 保存命令日志和修改记录。

### 4.2 P1 强烈建议交付

- 任务步骤时间线视图。
- 自动识别并优先读取关键配置文件，如 `package.json`、`pyproject.toml`、`Makefile`、`README`。
- 自定义忽略规则，支持 `.gitignore` 与额外 ignore 配置。
- 命令白名单与高风险命令拦截。
- 多模型配置，但同一时刻只使用一个模型。
- 用户偏好设置，如默认语言、默认审批模式、默认测试命令。
- 会话摘要，减少超长上下文成本。
- 错误信息聚合展示，便于定位失败原因。

### 4.3 P2 后续增强

- 符号级代码理解，如基于 LSP 或 tree-sitter 的定义/引用跳转。
- 浏览器自动化验证，用于前端页面调试。
- 项目长期记忆与规范沉淀。
- Tool 插件系统 / MCP 接入。
- 自动 commit 能力。
- 多模型路由。
- 多 Agent 协作执行。
- 本地知识库与向量检索。

## 5. 核心交互流程

### 5.1 主流程

1. 用户打开应用并选择本地工作区。
2. 用户输入任务，例如“修复当前失败的 pytest”。
3. Agent 读取项目基础上下文，如目录结构、关键配置和相关搜索结果。
4. Agent 生成简要计划，并进入工具执行循环。
5. Agent 执行搜索、文件读取、命令执行。
6. Agent 生成补丁并展示 diff。
7. 用户审批后应用补丁。
8. Agent 自动运行验证命令，如测试或构建。
9. 若验证通过，则任务完成。
10. 若验证失败，则 Agent 读取错误日志并尝试一轮修复。
11. 所有记录写入会话历史，便于后续恢复。

### 5.2 高风险操作流程

1. Agent 请求执行高风险操作，如写文件、删除文件、运行高风险命令。
2. 系统展示操作详情，包括目标文件、命令内容、影响范围。
3. 用户选择允许、拒绝或本次记住策略。
4. 系统根据用户选择执行或中止。

### 5.3 中断与恢复流程

1. 用户点击停止任务。
2. 系统终止当前模型调用与工具执行。
3. 已产生的日志、输出、diff 和中间状态被保存。
4. 用户可在后续重新打开会话继续执行。

## 6. 权限与安全策略

### 6.1 权限原则

采用“默认最小权限”原则，默认只允许 Agent 访问当前选定工作区及显式授权的工具。

### 6.2 文件系统权限

- 默认只允许读写当前工作区。
- 访问工作区外路径必须明确拦截或要求审批。
- 删除文件操作默认需要确认。
- 批量写入或跨目录写入属于高风险操作。

### 6.3 命令执行权限

- 默认允许低风险命令，如 `git status`、测试命令、只读搜索命令。
- 高风险命令需审批，如安装依赖、删除文件、网络下载、系统级修改。
- 命令必须支持超时与手动终止。
- 命令执行上下文默认绑定当前工作区。

### 6.4 网络权限

- MVP 默认关闭外部网络工具。
- 若后续开放模型侧联网或工具联网，必须在设置中显式开启。
- 所有联网行为需有用户可见提示。

### 6.5 数据与隐私

- 会话、日志、配置和历史记录默认保存在本机。
- 不将本地项目文件自动上传到除模型 API 外的其他远程服务。
- 用户需显式配置模型 API Key。
- 提供清理会话和本地数据能力。

### 6.6 审批策略

支持三种模式：

- 全审批：所有写操作与命令执行均需确认。
- 风险审批：仅高风险命令与写操作需确认。
- 宽松模式：低风险操作自动执行，高风险操作确认。

## 7. 非功能需求

### 7.1 性能

- 首次打开工作区后，目录可见时间不超过 3 秒。
- 常规全文搜索响应时间在中小型仓库中不超过 2 秒。
- 单次工具执行状态更新需实时流式显示，UI 感知延迟不超过 500ms。
- 50MB 以内单文件读取需支持分段加载，避免 UI 卡顿。

### 7.2 稳定性

- 工具执行失败、模型返回异常、命令超时等场景不得导致应用崩溃。
- 会话数据异常退出后应可恢复到最近一次可用状态。
- 写文件前必须具备可回滚基础，可通过 patch 或 Git 状态恢复。

### 7.3 可维护性

- 前端、Runtime、Tool 层解耦。
- Tool 接口结构化定义，便于后续扩展。
- 存储层采用可迁移 schema。
- Provider 接口可替换，便于接入 OpenAI / Anthropic。

### 7.4 可观察性

- 每次任务都有唯一任务 ID。
- 记录模型调用、工具调用、耗时、错误码、用户审批结果。
- 本地日志可导出，用于排障。

### 7.5 兼容性

- 首个版本优先支持 Windows 和 macOS，本地项目语言至少覆盖 Python / Node.js 常见仓库。
- 前端框架采用 `Tauri + React`。
- Runtime 采用 `Python`。

## 9. 里程碑与验收标准

### 9.1 M1：可用的本地聊天与工具壳

范围：

- 工作区选择
- 基础聊天界面
- 会话保存
- 文件搜索与文件读取
- 基础模型接入

验收标准：

- 用户可打开本地项目并发起会话。
- 可完成文件搜索与片段读取。
- 模型回复可流式显示。
- 历史会话可查看和恢复。

### 9.2 M2：闭环式代码修改

范围：

- Tool 调用循环
- patch 生成与应用
- diff 展示与审批
- shell 命令执行
- Git 状态查看

验收标准：

- 用户可要求 Agent 修改本地代码。
- Agent 可展示结构化 diff。
- 用户确认后可写入文件。
- 可执行测试或构建命令并展示输出。
- Git diff 能反映本次变更。

### 9.3 M3：自动验证与错误恢复

范围：

- 自动验证命令
- 失败日志回灌
- 一轮自动修复
- 中断与恢复
- 超时与风险控制

验收标准：

- 针对测试失败场景，Agent 能完成“修改 -> 验证 -> 基于错误再修复”至少一轮。
- 用户可随时停止任务。
- 超时命令可被终止。
- 高风险操作遵循审批规则。

### 9.4 MVP 发布验收

满足以下条件即可认定 MVP 完成：

- 在至少 3 个真实本地仓库中完成 10 个以上实际任务测试。
- 其中至少 70% 任务可完成“搜索 -> 修改 -> 验证 -> 展示结果”的闭环。
- 写操作全部具备 diff 审批能力。
- 应用在长时间使用下无阻断式崩溃。
- 所有历史会话和命令日志可追溯。

## 10. 风险与取舍

### 10.1 上下文过载风险

风险：模型容易因为上下文过多而增加成本、降低准确率。

取舍：MVP 不做全仓 embedding 检索，优先采用 `ripgrep + 关键文件优先 + 最近访问缓存` 的精简策略。

### 10.2 代码修改不可靠风险

风险：模型可能改错位置、误删逻辑、破坏格式或引入新错误。

取舍：强制使用 patch/diff 审批，不允许黑箱式整文件覆盖。所有修改后必须优先接验证命令。

### 10.3 Shell 执行安全风险

风险：Agent 若可任意执行命令，容易产生误删、误安装、超范围操作。

取舍：采用工作区隔离、审批模式、命令超时和高风险命令拦截，不追求完全自动化。

### 10.4 架构过重风险

风险：过早引入多 Agent、工作流引擎、向量数据库会显著延长交付周期。

取舍：MVP 仅做单 Agent、单工作区、本地 SQLite 持久化，不引入 Redis、Temporal、Qdrant。

### 10.5 模型依赖风险

风险：模型能力、成本、延迟、接口兼容会影响产品稳定性。

取舍：抽象 Provider Adapter，但 MVP 初期仅稳定支持少量模型，不追求复杂路由。

### 10.6 用户信任风险

风险：若 Agent 的行为不可见、不可控，用户不会放心在真实项目中使用。

取舍：优先投入在执行过程可视化、diff 审批、日志与恢复能力，而不是更“智能”的黑箱自动化。
